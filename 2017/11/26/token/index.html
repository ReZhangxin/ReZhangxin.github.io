Cookie Session Token

发表于 2017-11-26 | 分类于 Node

Cookie

cookie是一个非常具体的东西，指的就是浏览器里面能永久存储的一种数据，仅仅是浏览器实现的一种数据存储功能。

cookie由服务器生成，发送给浏览器，浏览器把cookie以k-v形式保存在某个目录下的文本文件内，下一次
请求同一网站时会把该cookie发送给服务器。由于cookie存在客户端上，所以浏览器加入一些限制，
确保cookie不会被恶意使用，同时不会占据太多磁盘空间，所以每个域的cookie数量是有限的。

Session

session（会话）。相当于你和一个狗狗玩，你怎么知道和你玩的是二哈而不是泰迪？这个狗狗肯定有某种特征表明它就是二哈。

session也是类似的道理，服务器要知道当前发请求给自己的是谁。为了做这种区分，服务器要给每个客户端分配不同的“身份标识”，
服务器就知道这个请求来自于谁了。至于客户端怎么保存这个“身份标识”，可以有很多方式，对于浏览器客户端，大家默认采用cookie方式

服务器使用session把用户的信息临时白存在服务器上，用户离开网站后session会被销毁。这种用户信息存储方式相对cookie来说更安全
可session有一个缺陷：如果web服务器做了负载均衡，那么下一个操作请求到了另一台服务器的时候session会丢失.

Token

token（令牌），是用户的验证方式，最简单的token组成：

  • uid（用户唯一的身份标识）
  • time（当前时间的时间戳）
  • sign（签名，由token的前几位+盐以哈希算法压缩成一定长的十六进制字符串，可以防止恶意第三方拼接token请求服务器）

传统的身份验证

HTTP是一种无状态协议，就是不知道是谁访问应用（盲僧）。客户端使用用户名和密码通过了身份验证，不过下回这个客户端
再发送请求的时候，还要再验证一下。你给盲僧一个苹果，然后你再给他一个，他并不知道给他苹果的人都是你！

解决方法：当用户请求登录的时候，如果没有问题，

  • 我们在服务端生成一条记录，这个记录里可以说明一下登陆的用户是谁，

  • 然后把这条记录的ID号发送给客户端，

  • 客户端收到以后把这个ID号存储在Cookie里，下次这个用户再向服务器发送请求的时候，可以带着这个cookie，

  • 这样服务端会验证一个这个cookie里的信息，看看能不能在服务端找到对应的记录，

  • 如果可以，说明用户已经通过了身份验证，就把用户请求的数据返还给客户端。

上面说的就是session，我们需要在服务端储存为登陆的用户生成session，这些sessin可能存储在内存，磁盘，或者数据库里。我们可能需要在服务器端定期的去清理过期的session

基于Token的身份验证

使用基于Token的身份验证方法，在服务端不需要存储用户的登录记录。

  • 客户端使用用户名和密码请求登录
  • 服务端收到请求，去验证用户名和密码
  • 验证成功后，服务端会签发一个Token，再把这个Token发送给客户端
  • 客户端收到Token以后可以把它存储起来，比如放在Cookie里或者LocalStorage里面
  • 客户端每次向服务端请求资源的时候需要带着服务端签发的Token
  • 服务端收到请求，然后去验证客户端请求里面带着的Token，如果验证成功，就向客户端返回请求的数据
